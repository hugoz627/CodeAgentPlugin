---
description: 并行扫描当前仓库或当前改动中的 bug、回归、边界条件和测试缺口
---

这个命令默认**不需要传参数**。

目标是直接基于**当前 diff、当前分支、当前任务上下文、当前计划文件**尽可能快地找出真实问题，而不是让用户先手工补范围。

默认上下文解析规则：
- 优先检查当前未提交 diff
- 如果当前 diff 很小，再扩展到被改动文件的直接依赖和调用链
- 如果当前任务已绑定计划文件，优先结合 `docs/superpowers/plans/` 中对应计划理解回归风险
- 只有在用户明确想聚焦某个文件、目录或子范围时，才把额外输入当作补充范围

适用场景：
- 怀疑当前改动里藏着 bug
- 想在提交前做一轮高强度回归扫描
- 需要多工具并行从不同角度找问题
- 需要补齐测试缺口、边界条件和异常路径

执行要求：

1. **先确定排查范围**
   - 默认先扫当前未提交 diff
   - 再扩展到被改动文件的直接依赖、调用链和关键影响面
   - 如果用户额外指定了文件、目录或 diff 范围，再叠加该范围

2. **排查维度至少覆盖**
   - 正确性：逻辑错误、状态流转错误、条件分支遗漏
   - 边界条件：空值、零值、重复输入、异常返回、并发/时序问题
   - 回归风险：旧行为是否被破坏、兼容性是否变化
   - 测试缺口：缺少关键路径测试、异常路径测试、回归测试
   - 安全与稳定性：输入校验、敏感信息处理、资源泄漏、错误恢复

3. **如果宿主支持多工具并行分派**
   - 可以把不同维度分给不同工具并行扫描
   - 推荐分法：
     - 一个工具查逻辑正确性和边界条件
     - 一个工具查安全、稳定性和回归风险
     - 一个工具查测试缺口和可维护性
   - 当前工具不要空等，要继续做主线归纳和复核

4. **如果宿主不支持并行分派**
   - 也必须顺序完成上述维度
   - 不要只看一遍 diff 就草率下结论

5. **输出必须去重和归并**
   - 同一根因不要拆成几十条重复问题
   - 需要把相同根因按“问题簇”汇总

6. **输出格式必须可执行**
   - 优先列出阻断问题
   - 每个问题尽量附文件位置、触发方式、影响、建议修法

输出格式：

```markdown
# Bug Sweep 报告

## 阻断问题
1. ...

## 重要风险
1. ...

## 测试缺口
1. ...

## 建议优先修复顺序
1. ...
```
