---
description: 需求完成后做代码质量与计划合规性的完整审查
---

这个命令默认**不需要传参数**。

目标是直接基于**当前 diff、当前任务上下文、当前计划文件**执行完整收尾审查；只有在用户想额外限定审查范围时，才把额外输入当作补充约束。

默认上下文解析规则：
- 优先找出当前 diff 范围
- 优先在 `docs/superpowers/plans/` 中搜索最相关计划文件，再回退到 `docs/plans/`
- 如果没有计划文件，就把当前任务目标、需求文本和用户补充说明视为对照基准
- 如果当前上下文已经足够，不要停下来要求用户再传计划路径

目标：
- 检查代码质量、测试充分性、潜在回归、边界条件和架构问题
- 检查实现是否符合原始计划或需求
- 输出可以直接修复的明确问题列表，而不是空泛评价

执行要求：

1. 先收集上下文：
   - 找出当前 diff 范围
   - 找出对应计划文件；如果用户显式给了路径，优先使用；否则按默认计划发现规则自动定位
   - 如果没有计划文件，就把“需求文本 / 当前任务目标 / 用户补充说明”视为对照基准，不要因为缺计划就跳过检查

2. 审查必须覆盖两条线：
   - **代码质量线**：Bug、错误处理、边界条件、性能、安全、测试、可维护性
   - **计划合规线**：哪些需求已实现、哪些遗漏、哪些偏离、哪些是计划外新增

3. 如果宿主支持 subagent / 并行审查：
   - 并行跑两条线的审查
   - 一条聚焦代码质量
   - 一条聚焦计划合规
   - 审查结束后汇总

4. 如果宿主不支持并行 subagent：
   - 也必须顺序完成两条线
   - 不得因为缺少 subagent 就省略其中一条

5. 输出时按严重程度排序，至少分为：
   - 阻断问题
   - 重要问题
   - 一般建议

6. 每个问题尽量附：
   - 文件路径
   - 位置
   - 问题原因
   - 建议修法

7. 最终结论必须明确给出：
   - 是否可以视为收敛
   - 还剩哪些必须修
   - 是否建议继续下一轮修复 + 审查

输出格式：

```markdown
# 完整审查报告

## 一、代码质量审查
...

## 二、计划合规性审查
...

## 三、结论
- 当前状态：通过 / 需修复 / 需重大修改
- 必须处理的问题：
  1. ...
- 下一步建议：
  1. ...
```
