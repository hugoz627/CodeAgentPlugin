---
description: 对当前 diff 执行 review、fix、verify 循环，直到问题收敛
---

这个命令默认**不需要传参数**。

它适用于：代码已经写得差不多了，但你希望当前工具**基于当前 diff、当前计划和当前任务上下文**持续收敛问题直到能交付。

与 `/autopilot-plan` 的区别：
- `/autopilot-plan` 面向“从计划到实现到收尾”的完整闭环
- 这个命令面向“当前 diff 已存在，重点是把它稳定下来”

默认上下文解析规则：
- 优先识别当前未提交 diff 和当前分支相对基线的改动
- 优先在 `docs/superpowers/plans/` 中定位当前任务计划，再回退到 `docs/plans/`
- 如果当前上下文已足够，不要要求用户再补范围
- 只有在用户明确想缩小范围时，才把额外输入当作补充约束

执行要求：

1. **先识别当前工作面**
   - 当前未提交 diff
   - 当前分支相对基线的改动
   - 对应计划文件、需求目标或当前任务基准

2. **先做第一轮完整审查**
   - 优先执行 `/review-complete`
   - 如果宿主不支持直接调用命令，就完整执行等价流程

3. **对问题分级处理**
   - 阻断问题：必须立即修
   - 重要问题：优先修
   - 一般建议：评估收益后决定是否修

4. **进入收敛循环**
   - 修复问题
   - 运行最小必要验证：测试、lint、类型检查、构建或定向验证
   - 再做一轮 `/review-complete`
   - 如有必要，穿插 `/cross-review` 或 `/bug-sweep`
   - 持续循环，直到：
     - 阻断问题消失
     - 重要问题明显收敛
     - 新问题不再不断冒出

5. **不要无意义停下**
   - 除非遇到真实阻塞（权限、外部依赖、会触发付费或不可逆操作、需求冲突），否则继续推进
   - 不要停在“还剩几个问题，要不要我继续修”这种节点

6. **最终输出必须包含**
   - 修复了哪些问题
   - 哪些验证已通过
   - 哪些问题仍保留以及原因
   - 当前是否已经达到“可提交 / 可继续开发 / 仍需修复”

输出格式：

```markdown
# Stabilize Diff 结果

## 已修复问题
1. ...

## 已完成验证
1. ...

## 仍保留的问题
1. ...

## 当前状态
- 可提交 / 基本收敛 / 仍需继续修复
```
