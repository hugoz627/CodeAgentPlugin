---
description: 在当前 worktree 中按当前计划连续实现、验证、审查、修复直到收敛
---

这个命令默认**不需要传参数**。

目标不是再让用户补一遍计划路径或需求描述，而是直接基于**当前 worktree、当前任务上下文、当前计划文件**推进完整闭环，并且**不中途无意义停下**。

默认上下文解析规则：
- 优先使用当前 worktree
- 优先读取当前仓库固定计划目录 `docs/superpowers/plans/`
- 如果该目录下只有一个计划文件，直接使用它
- 如果有多个计划文件，优先选择与当前分支名、当前任务名、最近修改时间最匹配的那个
- 如果已经能从当前任务上下文中定位到计划，不要停下来要求用户再传路径
- 只有在确实找不到可执行计划时，才回退到当前任务目标、最近需求说明和当前 diff 作为执行基准

核心要求：
- 必须在 **git worktree** 中开发；如果当前已经在正确 worktree 中，就直接复用，不要重复创建
- 执行时遵循 superpowers 的计划执行链路：`using-git-worktrees` -> `writing-plans` 产出的实现计划 -> `subagent-driven-development` 或 `executing-plans`
- 能用 subagent 的地方就用 subagent，但不要把关键路径完全外包后自己空等
- 除非遇到真正硬阻塞，否则不要中途反复问用户“要不要继续”
- 实现完成后，必须进入 **review -> fix -> review** 循环，直到问题收敛

执行流程：

1. **定位当前执行基准**
   - 识别当前是否已经位于 worktree 中
   - 在 `docs/superpowers/plans/` 中定位当前任务对应的实现计划
   - 读取计划后先做一次批判性检查：是否缺步骤、是否有明显冲突、是否需要补最小澄清
   - 如果计划可执行，直接进入实现，不要停在“计划已找到，是否开始”这种无效节点

2. **确保 worktree 环境正确**
   - 如果当前就在目标 worktree 中，直接继续
   - 如果当前不在 worktree 中，再按 superpowers 的 worktree 规则创建并切换
   - 后续所有实现、测试、审查都在同一个 worktree 中完成

3. **按计划持续推进实现**
   - 主代理负责关键路径推进
   - 可并行、低耦合、不会阻塞当前下一步的子任务，交给 subagent 或其他开发工具
   - 每完成一个关键阶段就做局部验证，不要等到最后一起爆炸
   - 如果当前计划天然适合 stay-in-session 模式，优先按 `subagent-driven-development` 的方式推进
   - 如果当前场景更适合独立批次执行，也可按 `executing-plans` 的节奏推进，但不要人为增加等待

4. **验证实现**
   - 运行与当前项目相匹配的测试、lint、类型检查、构建或最小验证命令
   - 如果失败，先定位并修复，再继续，不要带病进入下一阶段
   - 任何“已完成”“已修复”“已通过”的表述，都必须有新鲜验证证据支撑

5. **进入收尾审查循环**
   - 实现完成后，执行 `/review-complete`
   - 如果宿主不支持直接调用命令，则完整执行与 `review-complete` 等价的流程
   - 对审查发现的问题进行修复
   - 修复后重新验证
   - 再次执行 `/review-complete`
   - 一直循环，直到：
     - 阻断问题消失
     - 重要问题基本收敛
     - 新问题不再持续冒出

6. **只在这些情况下停下来问用户**
   - 缺少必须的访问权限、密钥、外部系统信息
   - 会触发真实付费、上线、资源创建或不可逆操作
   - 计划与需求本身互相冲突，继续实现会高概率做错
   - 发现仓库中有异常变更，无法安全判断是否应继续
   - 当前仓库里根本找不到可执行计划，而且从现有上下文也无法可靠推断

7. **最终交付必须包含**
   - 使用的 worktree 路径
   - 实际采用的计划文件路径
   - 改动文件列表
   - 执行过的验证命令与结果
   - 最后一轮审查结论
   - 仍未解决的风险（如果有）

禁止事项：
- 不要要求用户每次传计划路径
- 不要停在“计划已经写好，是否继续”这种没有价值的节点
- 不要只实现不验证
- 不要只 review 不修复
- 不要 review 一次就结束
