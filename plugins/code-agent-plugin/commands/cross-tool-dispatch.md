---
description: 基于当前任务和当前计划拆分低耦合子任务，并分派给不同开发工具并行执行或审查
---

这个命令默认**不需要传参数**。

它的目标不是让用户再补一个分派 prompt，而是直接基于**当前任务上下文、当前 diff、当前计划文件、当前 worktree**完成拆分、分派、汇总和继续推进。

默认上下文解析规则：
- 优先读取当前仓库固定计划目录 `docs/superpowers/plans/`
- 结合当前分支、当前 diff、当前未完成任务和最近计划文件定位分派基准
- 如果已有明确的当前任务或当前计划，不要停下来要求用户补参数
- 只有在当前上下文不足以安全拆分任务时，才要求最小必要澄清

分派原则：

1. **先判断是否值得分派**
   - 只有在子任务之间**低耦合、可并行、输入输出边界清晰**时才分派
   - 如果任务存在强顺序依赖，就不要为了并行而并行

2. **保持主线在当前工具手里**
   - 当前工具负责关键路径推进和结果集成
   - 其他工具负责非阻塞的实现、审查、找 bug、UI、重构或专项分析
   - 不要把所有事都分派出去，然后主代理只负责等待

3. **按当前计划拆任务，不要凭空发散**
   - 优先从当前实现计划中抽取可独立执行的 task、模块或验证面
   - 如果计划没写成 task，也要先把当前工作拆成明确子任务，再分派
   - 分派目标必须是可交付、可验证、可汇总的具体工作项

4. **按强项分配**
   - `Claude`：架构、安全、复杂推理、计划合规审查
   - `Codex`：快速实现、重构、聚焦修复、工程执行
   - `Gemini`：前端、UI、创意界面、多模态相关任务

5. **优先复用现有 multi-agent-dispatch 能力**
   - 如果当前环境能使用本仓库的 `multi-agent-dispatch` 能力，优先使用它
   - 不要直连底层 `acpx`
   - 通过统一包装脚本完成路由、日志、超时和会话管理
   - 超时要显式设置：
     - 读审查类任务：`600`
     - 常规实现：`1800`
     - 大任务 / 长测试：`3600`

6. **建议执行流程**
   - 识别当前计划与当前 diff
   - 列出候选子任务
   - 标记哪些可以并行
   - 选择每个子任务的最合适工具
   - 发起分派
   - 当前工具继续做关键路径工作，不要空等
   - 汇总所有结果
   - 处理冲突、重复实现、结论不一致
   - 再做一次本地验证

7. **输出必须包含**
   - 当前采用的计划文件或任务基准
   - 子任务拆分结果
   - 每个子任务分派给谁
   - 为什么这样分
   - 哪些结果已合并
   - 哪些冲突还需要处理
   - 下一步主线动作

禁止事项：
- 不要要求用户每次手工补分派参数
- 不要分派互相强耦合的任务
- 不要让多个工具同时改同一小块代码而没有边界
- 不要只分派不汇总
- 不要把“review”和“实现”混成同一个模糊任务扔出去
