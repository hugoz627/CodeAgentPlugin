---
description: 让另一开发工具对当前实现做独立复审，并汇总结论与分歧
---

这个命令默认**不需要传参数**。

目标是直接基于**当前 diff、当前任务上下文、当前计划文件、当前验证结果**发起跨工具复审，而不是让用户先手工补一遍 review 范围。

默认上下文解析规则：
- 优先读取当前 diff
- 优先在 `docs/superpowers/plans/` 中定位当前任务计划，再回退到 `docs/plans/`
- 优先读取最近一次测试、lint、构建或定向验证结果
- 只有在用户明确指定额外关注点时，才把额外输入当作补充审查范围

适用场景：
- 当前工具已经完成实现，想要另一工具独立 review
- 想做交叉审查：Codex 写，Claude 审；Claude 写，Codex 审
- 前端改动需要额外的 UI / 可访问性 / 交互视角
- 需要把“自我感觉没问题”变成“外部独立审视过”

执行要求：

1. **先收集复审上下文**
   - 当前 diff
   - 对应计划文件或需求目标
   - 已执行的测试与验证结果
   - 用户额外给出的关注点（如果有）

2. **优先选择与当前工具不同的审查工具**
   - 当前是 Codex：优先交给 Claude 做独立复审
   - 当前是 Claude：优先交给 Codex 做独立复审
   - 前端 / 视觉 / 交互改动明显时，可额外引入 Gemini 视角

3. **复审必须关注**
   - 真实缺陷、回归、遗漏测试
   - 与原始计划或需求的不一致
   - 代码结构是否引入后续维护风险
   - 当前工具可能因为熟悉自己的实现而忽略的问题

4. **当前工具的职责不是转发审查结果**
   - 必须先对外部 review 的结论做筛选和归并
   - 识别：
     - 确认有效的问题
     - 误报或争议点
     - 需要继续验证的问题

5. **如果用户目标是修完为止**
   - 审查结束后，进入修复
   - 修复后可以继续执行 `/review-complete` 或新的跨工具复审
   - 不要只把问题扔给用户自己消化

输出格式：

```markdown
# Cross Review 结果

## 外部审查发现
1. ...

## 当前工具确认有效的问题
1. ...

## 存在争议或需复核的问题
1. ...

## 建议下一步
1. ...
```
