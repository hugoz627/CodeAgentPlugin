# Validation Cases

用这些样例检查 skill 是否真的在减少偏差，而不是只是把 prompt 变长。

## Case 1: 目标已明确

### 输入

“帮我把当前登录接口的 rate limit 补齐，并写一个 `/goal`。验收是新增中间件、测试通过、不要改动注册流程。”

### 预期

- 不应进入多轮脑暴
- 应快速补出 proof，例如具体测试或命令类型
- 应产出精简 contract 和最终 `/goal`

## Case 2: 目标模糊，但上下文强

### 输入

“给当前任务写个 `/goal`。”

### 前提

- 当前 diff 明显集中在 `skills/goal-contract/`
- `docs/superpowers/plans/` 已有相关计划文件

### 预期

- 应先读取仓库上下文
- 应提出候选推断，例如本次交付物是否是新增一个共享 skill
- 若关键槽位仍缺失，再进入脑暴

## Case 3: 目标模糊且上下文弱

### 输入

“我想把这块体验完善下，顺便帮我写个 `/goal`。”

### 预期

- 应识别为高歧义目标
- 应进入脑暴模式
- 每轮只问一个最关键问题
- 最多 5 轮后要么补齐关键信息，要么显式列出假设后生成

## Case 4: 宿主不明确

### 输入

“把这段需求整理成最佳实践 `/goal`。”

### 预期

- 如果无法可靠判断当前宿主
- 应先输出共享 contract
- 最后同时给出 Codex / Claude Code 两版 `/goal`

## 最低验收线

每个样例最终产物都应满足：

- 有单一主要目标
- 有明确 `done_when`
- 有明确 `proof`
- 有范围或限制信息
- 如存在推断，已显式标注
- 简单任务不会机械输出所有内部字段
