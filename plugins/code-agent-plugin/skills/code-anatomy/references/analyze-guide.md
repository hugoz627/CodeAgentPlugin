# 源码分析方法论

## 1. 探索策略

### 1.1 第一轮：全局扫描
- 使用 Agent(subagent_type=Explore, thoroughness=very thorough) 探索整个项目
- 关注点：顶层目录、package.json/Cargo.toml/go.mod 等配置、README
- 输出：项目技术栈、语言、框架、构建工具的初步判断

### 1.2 第二轮：入口追踪
- 找到入口文件（main、index、app 等）
- 从入口开始，追踪 import/require 链
- 绘制初始模块依赖关系

### 1.3 第三轮：核心模块深读
- 对每个核心模块读取关键文件（前 200 行即可判断职责）
- 关注：类型定义文件、接口文件、核心类/函数
- 记录关键类型签名和导出 API

### 1.4 第四轮：数据流追踪
- 追踪一次完整的用户请求从输入到输出的流程
- 记录数据经过的每个模块和转换
- 识别核心循环（event loop、request cycle 等）

## 2. 章节规划模板

### 推荐的 5-8 章结构

| 章序 | 主题 | 适用于 | 内容要点 |
|------|------|--------|---------|
| 1 | 全景概览 | 所有项目 | 定位、技术栈、目录结构、模块关系、设计原则 |
| 2 | 启动/入口 | 所有项目 | Boot sequence、初始化流程、配置加载、依赖注入 |
| 3 | 核心引擎 | 所有项目 | 主循环/请求处理/渲染引擎等最核心机制 |
| 4 | 数据模型 | 有复杂状态 | 数据结构、状态管理、持久化、缓存 |
| 5 | 功能层 | 有插件/工具 | 工具/中间件/插件的注册、执行、权限 |
| 6 | API/协议 | 有外部通信 | API 设计、协议实现、序列化、错误处理 |
| 7 | 扩展体系 | 可扩展项目 | 插件系统、hooks、配置、第三方集成 |
| 8 | 设计哲学 | 优秀项目 | 架构决策、trade-off、值得学习的模式 |

根据项目特点灵活调整。小项目 5 章即可，大项目可到 8 章。

## 3. 每章内容标准

### 必须包含
- **概述段**（200-300 字）：这个模块是什么，为什么存在，在整体中的位置
- **关键类型/接口**（至少 2 个代码块）：核心 type/interface/class 定义
- **执行流程**（至少 1 个流程图）：ASCII 或描述性的流程说明
- **核心函数分析**：关键函数的签名、参数、返回值、逻辑要点
- **设计要点**：为什么这样设计，有什么 trade-off

### 质量红线
- 不允许编造代码 — 所有代码必须来自实际源码
- 不允许省略 — 如果一个模块有 10 个关键函数，不能只介绍 3 个
- 不允许空洞描述 — "这个模块很重要" 不如 "这个模块通过 X 机制实现了 Y 功能"
- 流程图必须反映真实代码路径 — 不要画理想化的流程

### 可选包含
- 性能优化策略
- 错误处理机制
- 与其他模块的交互接口
- 历史演进和版本变化

## 4. 输出格式规范

### 文件命名
```
docs/chapters/
├── ch1-overview.md
├── ch2-bootstrap.md
├── ch3-core-engine.md
├── ch4-tools.md
├── ...
```

### Markdown 格式
- 标题层级：# 章标题 → ## 大节 → ### 小节 → #### 子节
- 代码块标注语言：```typescript, ```python, ```go 等
- 表格用于对比和汇总
- ASCII 图用于流程和架构
- 中英混合：解释用中文，技术术语保留英文

### 每章预期长度
- 最短：300 行 Markdown（小章节）
- 推荐：400-600 行（标准章节）
- 最长：800 行（核心章节）
