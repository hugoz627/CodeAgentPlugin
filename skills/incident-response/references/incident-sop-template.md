# 线上事故响应 SOP 模板

## 事故分级标准

| 级别 | 定义 | 响应时限 | 示例 |
|------|------|----------|------|
| P0 | 核心功能完全不可用，影响全部用户 | 15 分钟内响应 | 支付系统宕机、登录不可用 |
| P1 | 核心功能严重降级，影响大量用户 | 30 分钟内响应 | 下单成功率 < 50%、大范围报错 |
| P2 | 非核心功能不可用或性能严重下降 | 2 小时内响应 | 推荐功能失效、响应时间 > 5s |
| P3 | 轻微问题，有降级方案 | 下个工作日处理 | 边缘功能 bug、小范围数据异常 |

## 响应流程

### 第一步：发现与确认（0~5 分钟）

1. 确认告警来源：监控告警 / 用户反馈 / 自主发现
2. 初步判断影响范围：影响哪些服务、哪些用户、从什么时候开始
3. 定级：P0 / P1 / P2 / P3
4. 通知相关人员（P0/P1 需要立即拉群）

### 第二步：止损（5~30 分钟，P0/P1）

优先止损，再排查根因。常用止损手段：

- **回滚**：如最近有发布，立即回滚到上一版本
- **限流/降级**：关闭问题功能的开关，返回默认值
- **扩容**：如果是容量问题，紧急扩容
- **切流**：将流量切换到备用服务或地区

### 第三步：根因排查

1. 查看错误日志：
   ```bash
   grep -iE 'error|exception|fatal' {log_path} | tail -100
   ```

2. 查看请求量和错误率变化（监控曲线）

3. 关联最近变更：
   - 最近 24 小时有无代码发布？
   - 有无配置变更？
   - 有无基础设施变更（扩缩容、机器迁移等）？

4. 缩小范围：
   - 问题是所有请求还是特定参数？
   - 问题是所有机器还是部分机器？
   - 问题是持续存在还是偶发？

### 第四步：修复与验证

1. 提出修复方案，评估风险
2. 在测试环境验证
3. 上线修复
4. 观察监控指标恢复正常
5. 通知相关人员：问题已解决

### 第五步：事后复盘（24~48 小时内）

生成复盘报告，包含以下内容：

```markdown
## 事故复盘：{标题}

**事故时间**：{开始时间} ~ {恢复时间}，共 {时长}
**事故级别**：P{级别}
**影响范围**：{受影响用户数/功能/服务}

### 事故经过

{时间线，按时间顺序描述发现、响应、止损、修复的关键节点}

### 根本原因

{导致事故的直接原因和根本原因}

### 为什么没有提前发现

{监控、测试、代码审查等哪些环节没有拦截住}

### 改进措施

| 措施 | 负责人 | 截止日期 | 优先级 |
|------|--------|----------|--------|
| {具体措施} | {人名} | {日期} | P{级别} |

### 经验教训

{本次事故的关键教训，帮助团队避免类似问题}
```
